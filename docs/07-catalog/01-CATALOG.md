# Catalog

## Overview

The catalog realm is the in-game shop. It exposes a hierarchical tree of pages
to the client, where each page carries a set of purchasable offers. The server
handles real-time page navigation, offer display, purchase execution, gift
sending, and voucher redemption. Every management action is mirrored through
both an HTTP REST API and a CLI command set.

The realm lives under `pkg/catalog/` and follows the project's standard
hexagonal layout:

```
pkg/catalog/
  domain/             – entities, repository interface, domain errors
  application/        – Service, VoucherService, Redis cache helpers
  infrastructure/
    model/            – GORM ORM models
    store/            – PostgreSQL repository implementation
    migration/        – up/down schema migrations
    seed/             – bootstrap data seeds
  adapter/
    realtime/         – WebSocket packet handler (Runtime + dispatch)
    httpapi/          – REST route handlers and OpenAPI specs
    command/          – Cobra CLI subcommands
  packet/             – packet-ID constants and encoder types
```

## Architecture

The catalog service is wired during `serve` startup inside
`core/cli/serve_economy.go`. Dependency injection is explicit: the HTTP module,
WebSocket runtime, and CLI tree each receive the same application service
instance. Redis caching is configured as an optional post-construction step.

```
Config / Redis
      │
      ▼
catalogapp.Service ←── catalogstore.Repository ←── PostgreSQL
      │
      ├── adapter/realtime.Runtime  (WebSocket)
      ├── adapter/httpapi           (REST)
      └── adapter/command           (CLI)
```

## Page Model

A `CatalogPage` is one node in the catalog navigation tree. It carries
display metadata, rendering hints (layout and asset lists), and access
controls.

| Field | Type | Description |
|-------|------|-------------|
| `ID` | int | Stable page identifier |
| `ParentID` | *int | Parent page ID; `nil` marks a root node |
| `Caption` | string | Display title shown in the navigation tree |
| `IconImage` | int | Sprite ID used for the page icon in the tree |
| `PageLayout` | string | Client layout template name (see [Layout Types](#layout-types)) |
| `Visible` | bool | Whether the page node is shown in the navigation tree |
| `Enabled` | bool | Whether purchasing is allowed on this page |
| `MinPermission` | string | Dotted permission key required to view the page; empty = no restriction |
| `ClubOnly` | bool | Requires an active club subscription |
| `OrderNum` | int | Sort position among siblings |
| `Images` | []string | Page image asset keys (see [Images and Texts](#images-and-texts)) |
| `Texts` | []string | Page text content blocks (see [Images and Texts](#images-and-texts)) |

### Tree Hierarchy

Pages form a tree. The root is a synthetic node with no database row — it is
generated by the index builder from all pages whose `ParentID` is `nil`. Any
page may have children.

```
Root (synthetic)
├── Category A  (Visible, no offers)
│   ├── Subcategory A1  (Visible, has offers)
│   └── Subcategory A2  (Visible, has offers)
└── Category B  (Visible, no offers)
    └── Subcategory B1  (Enabled=false → pageId=-1 in wire packet)
```

When a page has `Enabled = false`, the server sets `pageId = -1` in the index
node, which tells the client that navigation to that node is disabled. The
page still appears in the tree (if `Visible = true`) but cannot be opened.

### Images and Texts

Each page stores two pipe-delimited strings in the database that are split
into positional arrays before being sent over the wire:

- **`Images`** — page image asset keys. The client resolves each key against
  the game resource bundle to render the actual image. This corresponds to
  `page_strings_1` in the reference emulator database schema.

- **`Texts`** — page body text blocks. Each element may contain inline HTML
  markup (`<b>`, `<i>`, `<a href="event:catalog/open/...">` navigation links,
  colour tags). This corresponds to `page_strings_2` in the reference emulator
  database schema.

Both arrays are purely positional. The number of slots used and the meaning of
each slot is determined by the **layout type** of the page. The general
convention across all known layouts is:

| Index | Images slot | Texts slot |
|-------|-------------|------------|
| `[0]` | Headline / main banner image | Primary body text (HTML supported) |
| `[1]` | Teaser / secondary image | Secondary / subtitle text |
| `[2]` | Special / extra image *(optional)* | Extended details text *(optional)* |

Empty strings are valid for any unused slot.

**Database storage** — both arrays are serialised as pipe-delimited (`|`)
strings. For example:
```
page_strings_1: catalog_frontpage_headline|catalog_teaser
page_strings_2: Welcome to the Shop!|Redeem a voucher here:
```
is stored as the two raw column values and then split on `|` before being
written into the wire packet.

## Layout Types

The `PageLayout` field is a free-form string that the client uses to select
the UI component responsible for rendering the page. The server uses it to
apply layout-conditional encoding rules when building the `catalog.page`
packet.

### Standard layouts

| Layout | Image slots | Text slots | Offer rendering |
|--------|-------------|------------|-----------------|
| `default_3x3` | [0] headline, [1] teaser | [0] description, [1] subtitle | All offers in a 3-column grid |
| `single_bundle` | [0] headline | [0] description | Single bundle item with full-width display |
| `recycler_info` | [0] main image, [1] extra image | [0] body text, [1] supplementary text | All offers |
| `trophies` | [0] headline | [0] description | Offers displayed as trophies |
| `spaces_new` | [0] headline, [1] teaser | [0] description | Offers displayed as room spaces |

### Special-behaviour layouts

These layouts trigger altered offer-count encoding on the server.

| Layout | Offer block | Extra data |
|--------|-------------|------------|
| `frontpage` | Always sent as count=0 (no offers rendered) | None |
| `club_buy` | Always sent as count=0 in `catalog.page`; offers arrive separately in `catalog.club_offers` (2405) | Compatibility alias for older HC shop seeds |
| `vip_buy` | Always sent as count=0 in `catalog.page`; offers arrive separately in `catalog.club_offers` (2405) | Canonical Nitro HC shop layout; footer remains `offerId=0`, `acceptSeasonCurrencyAsCredits=false`, `frontPageItems=0` |
| `frontpage4` | Always sent as count=0 | Front-page promotions list appended after the offer block (id, caption, image, pageLink, pageId) |

The `frontpage`/`club_buy`/`vip_buy` distinction means these pages are
**navigation or marketing hubs** — they display a curated layout without
exposing individual purchasable items through the normal offer list.

For Nitro-based HC pages, the page shell and the data widgets are split:

- `vip_buy` is the modern HC layout and uses `catalog.page` (804) only for the
  layout shell, then relies on `catalog.club_offers` (2405) for the actual
  membership offers.
- Pixel Server still treats `club_buy` as a compatibility alias so older seeded
  databases continue to receive the same club-offer side-channel packets.
- Pixel Server also emits `catalog.direct_sms_club_buy` (195) with an empty
  availability payload so Nitro receives the optional club-buy side-channel
  event exposed in the renderer protocol tables.
- `club_gifts` uses `catalog.page` (804) for the shell and `catalog.club_gift_info`
  (619) for the actual gift selector entries.
- The `catalog.page` footer should stay aligned with reference emulators:
  `offerId = 0`, `acceptSeasonCurrencyAsCredits = false`, and a trailing
  `frontPageItems` count (zero unless a frontpage-style payload is appended).

### Page type override

`RECENT_PURCHASES` is a page **type** (not a layout) stored in the database.
When the server detects this type it replaces the normal offer list with the
player's recent purchase history — the same offer wire structure is used but
the data comes from the player's purchase log.

Pixel Server currently maps `RECENT_PURCHASES` via the `CatalogPageType`
distinction inherited from the reference emulators. The layout field may be
any value; only the type value drives the encoding override.

## Access Control

Three independent gates protect catalog page access:

1. **`MinPermission`** — if non-empty, the player must hold the named
   permission (dotted key, e.g. `catalog.vip.view`). Evaluated by the
   permission resolver. Empty string allows all players.

2. **`ClubOnly`** — if true, the player must have an active club subscription.
   Club level is read from the player's `user.permissions` data.

3. **`Enabled`** — if false, the page is in the tree but cannot be opened.
   The index packet marks the node with `pageId = -1`.

All three conditions are evaluated on every `catalog.get_page` request.
Violation produces the relevant domain error and no page packet is sent.

## Caching

The catalog index and per-page offer lists are cached in Redis to avoid
database round-trips on every navigation request.

| Cache key | Content | TTL |
|-----------|---------|-----|
| `catalog:pages` | All `CatalogPage` rows (JSON) | 5 minutes |
| `catalog:offers:{pageID}` | All `CatalogOffer` rows for one page (JSON) | 5 minutes |

Cache is populated on the first miss and invalidated on any write mutation
(create, update, delete) to the affected entity. The TTL and key prefix are
configurable via `CacheConfig` at wiring time.

When Redis is not configured the service falls back to direct database reads
without error.
