# Protocol Code Generation

## Goals

1. `pkg/protocol` is **always in sync** with `vendor/pixel-protocol/spec/protocol.yaml`.
2. Every packet is a **typed Go struct** – no `map[string]interface{}` anywhere in the hot path.
3. Encode/decode logic is **generated, not hand-written** – the spec is the single source of truth.
4. Adding a new packet requires touching only two files: `protocol.yaml` (spec) and the corresponding service handler – nothing in between.

---

## Is it possible?

Yes. The spec is mature enough to drive full generation:

- Every packet has a fixed `id` (uint16), `direction` (c2s / s2c), `realm`, and an ordered `fields` array with typed entries.
- All used types map 1:1 to Go primitives or composite types already in `pkg/codec`:

| YAML type | Go type | Wire encoding |
|---|---|---|
| `boolean` | `bool` | uint8 (0/1) |
| `int32` | `int32` | big-endian int32 |
| `uint16` | `uint16` | big-endian uint16 |
| `uint32` | `uint32` | big-endian uint32 |
| `string` | `string` | uint16 len + UTF-8 bytes |
| `bytes` | `[]byte` | raw |
| `list<T>` | `[]T` | uint32 count + N×T |

Currently 922 packets are defined; the generator handles all of them.

## Generator design (`tools/protogen`)

```
tools/protogen/
  main.go           ← CLI entry-point (flags: -spec, -out, -package)
  loader.go         ← parses protocol.yaml → internal AST
  types.go          ← YAML type → Go type mapping
  struct_gen.go     ← emits packet struct declarations
  codec_gen.go      ← emits Encode / Decode methods
  router_gen.go     ← emits RouterTable with HeaderID constants
  handler_gen.go    ← emits handler interface stubs (skips if file exists)
  template/
    packet.go.tmpl
    router.go.tmpl
    handler.go.tmpl
```

### Internal AST (Go structs)

```go
type Spec struct {
    Packets struct {
        C2S []PacketDef `yaml:"c2s"`
        S2C []PacketDef `yaml:"s2c"`
    } `yaml:"packets"`
}

type PacketDef struct {
    ID        uint16     `yaml:"id"`
    Name      string     `yaml:"name"`
    Realm     string     `yaml:"realm"`
    Fields    []FieldDef `yaml:"fields"`
    Header    string     `yaml:"header"`
}

type FieldDef struct {
    Name        string `yaml:"name"`
    Type        string `yaml:"type"`
    Required    bool   `yaml:"required"`
}
```

### Emitted struct (example)

Input (spec):
```yaml
- id: 4000
  name: handshake.release_version
  direction: c2s
  fields:
    - name: releaseVersion
      type: string
      required: true
    - name: clientType
      type: string
      required: true
    - name: platform
      type: int32
      required: true
    - name: deviceCategory
      type: int32
      required: true
  header: RELEASE_VERSION
```

Output (`pkg/protocol/handshake.go` – generated section):

```go
// Code generated by protogen. DO NOT EDIT.

const HeaderReleaseVersion uint16 = 4000

// ReleaseVersionPacket is sent by the client immediately after WebSocket connection opens.
// Direction: c2s  Realm: handshake-security
type ReleaseVersionPacket struct {
    ReleaseVersion string
    ClientType     string
    Platform       int32
    DeviceCategory int32
}

func (p *ReleaseVersionPacket) HeaderID() uint16 { return HeaderReleaseVersion }

func (p *ReleaseVersionPacket) Encode(w *codec.Writer) error {
    w.WriteString(p.ReleaseVersion)
    w.WriteString(p.ClientType)
    w.WriteInt32(p.Platform)
    w.WriteInt32(p.DeviceCategory)
    return nil
}

func DecodeReleaseVersion(r *codec.Reader) (*ReleaseVersionPacket, error) {
    p := &ReleaseVersionPacket{}
    var err error
    if p.ReleaseVersion, err = r.ReadString(); err != nil { return nil, err }
    if p.ClientType,     err = r.ReadString(); err != nil { return nil, err }
    if p.Platform,       err = r.ReadInt32();  err != nil { return nil, err }
    if p.DeviceCategory, err = r.ReadInt32();  err != nil { return nil, err }
    return p, nil
}
```

### Router table (generated)

```go
// pkg/protocol/router.go – Code generated by protogen. DO NOT EDIT.

type HandlerFunc func(session *Session, r *codec.Reader) error

var C2SRouter = map[uint16]HandlerFunc{}

func RegisterC2SHandler(id uint16, h HandlerFunc) {
    C2SRouter[id] = h
}
```

### Handler stub generation

For each c2s packet, `handler_gen.go` emits a stub **once** into the appropriate service's `handlers/` directory. If the file already exists, it is skipped (so manual logic is preserved):

```go
// services/game/handlers/handshake_release_version.go
// Stub generated by protogen — safe to edit.
package handlers

import (
    "pixel-server/protocol"
    "pixel-server/services/game/session"
)

func HandleReleaseVersion(sess *session.Session, pkt *protocol.ReleaseVersionPacket) error {
    // TODO: implement
    return nil
}
```

## `list<T>` handling

`list<T>` is the only parametric type. The generator resolves the element type from the field annotation (e.g. `list<int32>`, `list<FurnitureItem>`).

For primitive element types, emission is straightforward. For composite element types referenced from the same spec (e.g. `FurnitureItem`), the generator emits a shared `types.go` file within the same package containing the composite struct and its own encode/decode methods.

## Integration with existing JavaScript generator

The existing `vendor/pixel-protocol/tools/generator/generate-artifacts.mjs` generates **Markdown documentation** from the same YAML. The Go `protogen` is a parallel artifact – it uses the same YAML source but targets a different output. Both generators run in separate pipelines:

```
Makefile:
  generate-docs:   cd vendor/pixel-protocol && npm run generate
  generate-go:     go run ./tools/protogen …
  generate:        generate-docs generate-go
```

No changes to the Markdown generator are needed.

---

## Validation guarantees

The generator fails hard (non-zero exit) if any of the following is violated:

- A field type is not in the supported type table.
- Two packets in the same direction share the same header ID.
- A field marked `required: true` has a Go zero-value type that is ambiguous (e.g. `int32` when 0 is a meaningful value) — emits a `// required` comment for reviewer attention, but does not block generation.
- A `list<T>` element type references a composite type not defined in the spec's `compositeTypes` section.

---

## Go generate invocation

Each package that contains generated files carries the directive:

```go
// pkg/protocol/doc.go
//go:generate go run ../../tools/protogen -spec ../../vendor/pixel-protocol/spec/protocol.yaml -out .
package protocol
```

Running `go generate ./pkg/protocol/` is sufficient to regenerate the entire protocol package.
